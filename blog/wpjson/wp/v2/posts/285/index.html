From Microsoft….<\/p>\n

\n

Source Analysis is similar in many ways to Microsoft Code Analysis (specifically FxCop), but there are some important distinctions. FxCop performs its analysis on compiled binaries, while Source Analysis analyzes the source code directly. For this reason, Code Analysis focuses more on the design of the code, while Source Analysis focuses on layout, readability and documentation. Most of that information is stripped away during the compilation process, and thus cannot be analyzed by FxCop. <\/p>\n

The ultimate goal of Source Analysis is to allow you to produce elegant, consistent code that your team members and others who view your code will find highly readable. In order to accomplish this, Source Analysis does not allow its rules to be very configurable. Source Analysis takes a one-size-fits-all approach to code style, layout, and readability rules.<\/strong><\/font> It is highly likely that you will not agree with all of the rules and may even find some of the rules annoying at first! However, the majority of teams using this tool within Microsoft have found that after a short adjustment period, they came to appreciate the rules enforced by Source Analysis, and even began to find it difficult to read code not written in this style.  <\/p>\n

Source Analysis comes with a set of default rules analyzers covering approximately 200 best practice rules. These rules are full compatible with the default layout settings in Visual Studio 2005 and Visual Studio 2008.<\/p>\n<\/blockquote>\n

I’ve downloaded it and tried it on a project I was working on for home. <\/p>\n

In a 428 line CS file, I had only 148 violations. :)<\/p>\n

<\/p>\n

A few of them I’m a bit surprised by I suppose, like the requirement that function calls must be prefixed with this<\/strong>.  <\/p>\n

“A comment may not be placed within the bracketed statement” and “A line may only contain a single statement”:<\/p>\n

\n
  } \/\/ catch
<\/span>catch <\/span>{ }
} \/\/ attachment loop\n<\/span><\/pre>\n<\/blockquote>\n
<\/a><\/p>\n
OK, I’ll give them the catch is a bit odd (my code just gobbles any exceptions), but I don’t know that I understand why I can’t put a comment after a bracket (it’s just marking the end of a block).<\/p>\n
\n
“All using directives must be placed inside of the namespace.”<\/p>\n<\/blockquote>\n
Eh? Why?<\/p>\n
\n
“Variable names must not start with m_”<\/p>\n<\/blockquote>\n
Got that one wrong, I occasionally use s_ to indicate static instance fields. It’s misfiring on that one.<\/p>\n
\n
“Field names must not start with an underscore.”<\/p>\n<\/blockquote>\n
Yeah, OK. No thanks. I like doing that and I’m not about to switch to prefixing everything with “this.” instead of an underscore. Bzzzt.<\/p>\n
\n
“An opening curly bracket must not be followed by a blank line.”<\/p>\n<\/blockquote>\n
Oh my heavens! A blank line after a curly bracket?\n<\/p>\n
Seriously though — if you’re wanting a rigorous source code formatting analysis tool, check this<\/a> out. If you buy into all of their rules — it’s all good. If you don’t follow all of Microsoft’s C# conventions (some which I wasn’t aware of), then you may be in for a rude awakening.\n<\/p>\n
Unfortunately, the thing that I’d like to have as a feature would be for it to optionally FIX the code to match the rules. Especially the easy stuff like blank lines and curly braces.
