I’ve created a reasonably simple, yet multi-technology (and discipline) demonstration using Silverlight for the user interface and ASP.NET as the back-end. The demonstration uses:<\/p>\n

    \n
  • Silverlight 2.0 <\/li>\n
  • Data binding <\/li>\n
  • Delayed downloading of images <\/li>\n
  • “Web services” <\/li>\n
  • ASP.NET <\/li>\n
  • LINQ <\/li>\n
  • Extension Methods <\/li>\n
  • Value Converters <\/li>\n
  • XAML <\/li>\n
  • IHttpAsyncHandler <\/li>\n
  • HttpWebRequest <\/li>\n
  • Google’s Weather Web Service (more of a weather XML end point) <\/li>\n
  • and more!!! :)<\/li>\n<\/ul>\n

    The Silverlight Weather control is a UserControl with a few bound TextBlocks, an Image, and a few rectangles, and the data input controls.<\/p>\n

    Standard Silverlight data binding … grabbing the CurrentWeather’s Location property (which in the example in the screen shot is Madison, WI). What I continue to find is that Silverlight does not allow data binding to the UserControl itself — which is a common trick I use in WPF all the time. I don’t know if it’s a bug or a feature though. So, instead of<\/strong> simple code like this:<\/p>\n
    public <\/span>Page()\n{\n    InitializeComponent();\n    this<\/span>.Loaded += new <\/span>RoutedEventHandler<\/span>(Page_Loaded);\n}\n\nvoid <\/span>Page_Loaded(object <\/span>sender, RoutedEventArgs <\/span>e)\n{\n    this.DataContext = this<\/span>;\n}\n<\/pre>\n
    I’m forced to rely on a secondary layer for binding:<\/p>\n
    public <\/span>Page()
    {
        InitializeComponent();
        this<\/span>.Loaded += new<\/span>RoutedEventHandler<\/span>(Page_Loaded);
        _binding = new<\/span>PageDataBinding<\/span>();
    }

    void <\/span>Page_Loaded(object<\/span>sender, RoutedEventArgs <\/span>e)
    {
        LayoutRoot.DataContext = _binding;
    }
    <\/pre>\n<\/p>\n
    This technique adds what isn’t a terribly useful layer for a project like this, but it shouldn’t be necessary. To work around the issue, I created a new class, called PageDataBinding which has the few properties I wanted to expose to the user interface:<\/p>\n
    public class <\/span>PageDataBinding<\/span>: INotifyPropertyChanged\n<\/span>{\n    private <\/span>WeatherCondition <\/span>_currentWeather;\n\n    public <\/span>WeatherCondition <\/span>CurrentWeather\n    {\n        get <\/span>{ return <\/span>_currentWeather; }\n        set\n        <\/span>{\n            if <\/span>(_currentWeather != value<\/span>)\n            {\n                _currentWeather = value<\/span>;\n                RaisePropertyChanged(\"CurrentWeather\"<\/span>);\n            }\n        }\n    }\n\n    #region <\/span>INotifyPropertyChanged Members\n\n    protected void <\/span>RaisePropertyChanged(string <\/span>name)\n    {\n        if <\/span>(PropertyChanged != null<\/span>)\n        {\n            PropertyChanged(this<\/span>, new <\/span>PropertyChangedEventArgs<\/span>(name));\n        }\n    }\n    public event <\/span>PropertyChangedEventHandler <\/span>PropertyChanged;\n\n    #endregion\n    \n<\/span>}<\/pre>\n
    <\/a><\/p>\n
    Above you should see the CurrentWeather property which I’ve used in the Silverlight UI. In addition to the typical button for activating the control, I’ve made the Enter key also trigger the weather fetching. <\/p>\n
    private void <\/span>txtLocation_KeyDown(object <\/span>sender, KeyEventArgs <\/span>e)\n{\n    if <\/span>(e.Key == Key<\/span>.Enter)\n    {\n        FetchWeather();\n        e.Handled = true<\/span>;\n    }\n}\n<\/pre>\n
    <\/a>Above, after (starting the process of) fetching the weather, I’ve set the Handled property to true, which indicates no further processing of the key is needed.<\/p>\n
    I wrapped the basic request to Google into a class called Weather. It’s the job of the Weather class to package the request and submit it to my exposed web service. Google’s web servers do not have any of the cross domain permission files needed by Silverlight for it to directly access them; so, I created a proxy web service just for the purpose. <\/p>\n
    Using the HttpWebRequest object is quite simple:<\/p>\n
    public void <\/span>Download()\n{\n    try\n    <\/span>{\n        HttpWebRequest <\/span>request = WebRequest<\/span>.Create(\n            new <\/span>Uri<\/span>(Application<\/span>.Current.Host.Source,\n                string<\/span>.Format(\n                \"..\/GetWeather.ashx?location={0}\"<\/span>, _location))) as <\/span>HttpWebRequest<\/span>;\n        request.Method = \"GET\"<\/span>;\n        this<\/span>._responseResult = \n            request.BeginGetResponse(new <\/span>AsyncCallback<\/span>(RequestCallback), request);\n    }\n    catch <\/span>(Exception <\/span>ex)\n    {\n        Debug<\/span>.WriteLine(ex);\n    }\n}\n<\/pre>\n
    <\/a>I’ve just made the request to my web server and passed the location parameter directly. If it’s gibberish, Google just returns an error, so I don’t bother with any special processing (and in the end, it’s just a demonstration).<\/p>\n
    When the response is received, the RequestCallback method is called:<\/p>\n
    private void <\/span>RequestCallback(IAsyncResult <\/span>result)\n{\n    HttpWebRequest <\/span>request = result.AsyncState as <\/span>HttpWebRequest<\/span>;\n\n    HttpWebResponse <\/span>response = (HttpWebResponse<\/span>) request.EndGetResponse(result);\n\n    Stream <\/span>s = response.GetResponseStream();\n\n    StreamReader <\/span>reader = new <\/span>StreamReader<\/span>(s);\n    string <\/span>webResult = reader.ReadToEnd();\n    Debug<\/span>.WriteLine(webResult);\n    reader.Close();\n    response.Close();\n\n    if <\/span>(WeatherDataDownloaded != null<\/span>)\n    {\n        WeatherDataDownloaded(this<\/span>, new <\/span>WeatherDataDownloadedEventArgs<\/span>(webResult));\n    }\n}\n<\/pre>\n
    <\/a><\/p>\n
    Not necessarily production ready code, but it works for this demo. The result is returned, and gathered into a string. Once complete, the Weather class raises an event with the result to the host user interface. <\/p>\n
    I won’t replicate all the code here, but once the user interface event code is called, it begins to parse the XML data:<\/p>\n

    <\/a>As you can see, I use LINQ and an extension method I frequently use to obtain a named attribute from an element. The extension method, GetChildElementAttributeValue doesn’t do much really:<\/p>\n
    public static string <\/span>GetChildElementAttributeValue(\n    this <\/span>XElement <\/span>element, XName <\/span>elementName, XName <\/span>attributeName)\n{\n    XElement <\/span>subElement = element.Element(elementName);\n    if <\/span>(subElement != null<\/span>)\n    {\n        XAttribute <\/span>attr = subElement.Attribute(attributeName);\n        if <\/span>(attr != null<\/span>)\n        {\n            return <\/span>attr.Value;\n        }\n    }\n\n    return null<\/span>;\n}<\/pre>\n
    The primary function (of this function!) is to protect against null elements when trying to read an attribute value. <\/p>\n
    The attribute values are stored in a WeatherCondition object. Once the object has been created, and a little more work is done, the code must notify the UI’s bound objects. <\/p>\n
    WAIT<\/strong>! If the notification happens now (on the currently executing thread), the Silverlight runtime will absolutely throw an exception, just like WPF would. The current executing thread is not the user interface thread, so I’ve used the Dispatcher to execute the update on the user interface thread:<\/p>\n
    Dispatcher.BeginInvoke(delegate<\/span>()\n{\n    _binding.CurrentWeather = conditions.First<WeatherCondition<\/span>>();\n});\n<\/pre>\n

    The remainder of the Silverlight code is mostly basic plumbing and user interface manipulation. There should be more error trapping than there is … but, I’m leaving that for another day (or for you!).<\/p>\n
    Rather than introducing a true web service into the project, I instead created an IHttpAsyncHandler in ASP.NET. This way, the handler could efficiently make requests of the Google Weather API and return them, without blocking further requests. In an ideal world, I’d add caching and more error handling ….<\/p>\n

    I’ve also created an IValueConverter to take the path to an image and convert it to a bitmap image. <\/p>\n
    BitmapImage <\/span>bi = new <\/span>BitmapImage<\/span>();\nbi.UriSource = new <\/span>Uri<\/span>(\"\/Images\/\" <\/span>+ path, UriKind<\/span>.Relative);  \n<\/pre>\n
    The images are stored on the web server rather than being embedded directly into the XAP file. Note that for this to work, the images must be stored in the ClientBin\/Images folder, … not<\/strong> the \/Images folder of the web server (as the path is relative to the location of the Silverlight download, not to the web application).<\/p>\n
    One issue that I ran into that had me puzzled for a while. AG_E_NETWORK_ERROR. This error occurred when trying to directly download the images Google uses for weather icons. That error had me baffled, until I realized that Silverlight cannot download images from other domains, unless they have the cross domain XML files properly configured for remote access. Google, sadly, does not have them, so I created my own images using Expression Design. The original file can be found in the download if you’re interested (as a series of layers). <\/p>\n
